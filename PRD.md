## ---

**📝 油猴腳本「沉浸式翻譯」第一階段：需求書 (BDD 模式)**

## **📌 1\. 願景與核心價值**

作為一個習慣閱讀外文網站的使用者，我希望在閱讀網頁時，能透過「點擊懸浮球」將網頁內的原生段落翻譯為中文，並以**原文與譯文上下對照**的方式呈現，以便於我能流暢且無縫地理解網頁內容。

## ---

**📜 2\. 業務規則與實例（Examples）**

## **規則一：原文與譯文雙語對照**

**目的**：釐清段落翻譯後的 DOM 狀態與排版規則。

* **實例 1.1：純文字段落的翻譯對照**  
  * **Given** 系統中已配置有效的 LLM API。  
  * **And** 網頁存在一個未翻譯的段落節點 Node\_A。  
  * **When** Node\_A 觸發翻譯並成功取得翻譯結果 "你好"。  
  * **Then** Node\_A 原生節點保持原樣不變。  
  * **And** 系統在 Node\_A 的正下方，新增一個包含 "你好" 的獨立對照節點。

## ---

**規則二：極簡標籤處理與安全失敗（Fail-safe）**

**目的**：定義複雜網頁結構的邊界，以及面對 LLM 輸出不穩定時的系統容錯防線。

* **實例 2.1：忽略複雜 HTML 標籤（降低首輪複雜度）**  
  * **Given** 待翻譯段落節點為 \<p\>Read \<a href='x'\>docs\</a\> now.\</p\>。  
  * **When** 系統準備發送請求給 LLM。  
  * **Then** 系統直接提取該節點的 innerText（即 "Read docs now."），不對超連結進行任何保留或替換處理。  
* **實例 2.2：安全失敗機制（防 LLM 破壞格式）**  
  * **Given** Node\_B 正在發起翻譯請求。  
  * **When** LLM API 回傳錯誤、超時，或是回傳了無法解析的亂碼。  
  * **Then** Node\_B 應取消載入中狀態，並直接顯示原文，不做任何強制譯文渲染。

## ---

**規則三：懸浮球位置持久化**

**目的**：使用狀態轉移（State Transition）來描述 UI 元件的記憶能力，避免寫成依賴 UI 拖曳流程的自動化腳本\[agile3uncles\]。

| 起始狀態 | 觸發事件 | 目標狀態 | 備註（隱含業務規則） |
| :---- | :---- | :---- | :---- |
| **預設狀態** | 使用者初次安裝並開啟網頁 | 懸浮球位於網頁**右下角** | 預設不遮擋主流網頁導覽。 |
| **預設狀態** | 使用者移動懸浮球至位置 (X1, Y1) | 懸浮球停留在 **自訂位置 (X1, Y1)** | 避免與網頁上原生的客服視窗或按鈕重疊。 |
| **自訂位置 (X1, Y1)** | 使用者重新整理網頁（F5） | 懸浮球保持在 **自訂位置 (X1, Y1)** | 該座標狀態必須成功寫入 GM\_setValue 達成跨分頁持久化。 |

## ---

**🚧 3\. 需求邊界與排除範圍 (Out of Scope)**

為了確保首輪 MVP 的輕量與高交付率，以下需求在此階段**明確不支援**：

1. **不支援** PDF、EPUB 或 YouTube 字幕等特殊媒體的翻譯。  
2. **不支援** 複雜的 HTML 標籤還原與高亮（如點擊譯文中的超連結跳轉）。  
3. **不支援** Stream 流式打字機輸出，API 採一次性完整返回（Non-stream）模式。  
4. **不支援** 瀏覽器本地的 IndexedDB 或是 SQLite 快取（每次點擊都將重新消耗 API Token）。
